在 C++14 中，```std::shared_timed_mutex``` 和 ```std::shared_lock``` 是并发编程领域的一对“黄金搭档”，它们共同实现了高效的读写锁（Reader-Writer Lock）机制，并支持超时控制。


这是 C++14 对 C++11 并发库最重要的补充之一，解决了“多读单写”场景下的性能瓶颈。

## 1. 核心概念

### 1.1. ```std::shared_timed_mutex```

+ 定义：一种互斥量，支持两种锁定模式：

  1. 独占模式 (Exclusive)：同一时刻只允许一个线程持有（用于写操作）
 
  2. 共享模式 (Shared)：同一时刻允许多个线程同时持有（用于读操作）

+ 特性：

  + 继承自 C++11 的超时能力。提供了 ```try_lock_for``` 和 ```try_lock_until``` 的共享/独占版本
 
  + 区别：相比 C++14 的 ```std::shared_mutex```，```std::shared_timed_mutex``` 额外支持带时间限制的锁定尝试
 
  + 注意：在某些实现中（如早期的 libstdc++），```shared_mutex``` 和 ```shared_timed_mutex``` 可能是同一个类，但在标准语义上，后者必须支持超时

### 1.2. ```std::shared_lock```

+ 定义：一个 RAII（资源获取即初始化）锁守卫类，专门用于管理 共享模式 的锁

+ 对应关系：

  + ```std::unique_lock``` ↔ 独占锁 (写锁) ↔ ```lock()```, ```unlock()```
 
  + ```std::shared_lock``` ↔ 共享锁 (读锁) ↔ ```lock_shared()```, ```unlock_shared()```

+ 行为

  + 当多个线程使用 ```std::shared_lock``` 锁定同一个 ```shared_timed_mutex``` 时，它们可以并发执行
 
  + 一旦有线程尝试获取独占锁（```unique_lock```），所有新的共享锁请求会被阻塞，直到独占锁释放

## 2. 代码实战：多读单写 + 超时

```c++
#include <iostream>
#include <thread>
#include <shared_mutex> // C++14 头文件
#include <chrono>
#include <vector>
#include <string>

class ConfigManager {
private:
    std::shared_timed_mutex mutex_;
    std::string config_data_;

public:
    // 【读操作】：多个线程可同时进行
    std::string get_config() {
        // 尝试获取共享锁，最多等待 100ms
        std::shared_lock<std::shared_timed_mutex> lock(
            mutex_, 
            std::chrono::milliseconds(100)
        );

        if (!lock.owns_lock()) {
            return "Error: Read timeout, could not acquire lock.";
        }

        // 模拟读取耗时
        std::this_thread::sleep_for(std::chrono::milliseconds(10));
        return config_data_;
    }

    // 【写操作】：互斥，排斥所有读操作
    bool update_config(const std::string& new_val) {
        // 尝试获取独占锁，最多等待 200ms
        std::unique_lock<std::shared_timed_mutex> lock(
            mutex_, 
            std::chrono::milliseconds(200)
        );

        if (!lock.owns_lock()) {
            return false; // 写入失败，超时
        }

        // 模拟写入耗时
        std::this_thread::sleep_for(std::chrono::milliseconds(50));
        config_data_ = new_val;
        return true;
    }
};

int main() {
    ConfigManager cm;
    // 初始值
    cm.update_config("Initial Value"); 

    std::vector<std::thread> workers;

    // 启动 5 个读线程
    for (int i = 0; i < 5; ++i) {
        workers.emplace_back([&cm, i]() {
            std::cout << "Reader " << i << ": " << cm.get_config() << "\n";
        });
    }

    // 启动 1 个写线程
    workers.emplace_back([&cm]() {
        if (cm.update_config("Updated Value")) {
            std::cout << "Writer: Update successful.\n";
        } else {
            std::cout << "Writer: Update failed (timeout).\n";
        }
    });

    for (auto& t : workers) t.join();
    return 0;
}
```

## 3. 面试核心考点

### 3.1. Q1: ```std::shared_lock``` 和 ```std::unique_lock``` 的本质区别是什么？

+ 答案：
  + 锁模式：shared_lock 调用底层的 lock_shared()，允许并发；unique_lock 调用 lock()，强制互斥。
  + 兼容性：
    + 多个 shared_lock 可以共存。
    + 一个 unique_lock 存在时，任何其他锁（shared 或 unique）都无法获取。
    + 已有 shared_lock 时，新的 shared_lock 可获取，但 unique_lock 需等待所有 shared 释放。
+ 适用场景：读多写少用 shared_lock 保护读路径；写操作必须用 unique_lock。

### 3.2. Q2: 为什么需要 ```std::shared_timed_mutex``` 而不仅仅是 ```std::shared_mutex```？

+ 答案：
  + 防止死锁/长时阻塞：在高并发系统中，如果写线程持有锁时间过长（或发生优先级反转），读线程可能会无限期阻塞。
  + 超时机制：shared_timed_mutex 允许线程在指定时间内尝试获取锁。如果超时，线程可以执行降级策略（如返回旧数据、报错或重试），提高系统的健壮性和响应性。
  + 接口差异：std::shared_mutex (C++14 基础版) 不提供 try_lock_for 等超时接口，而 shared_timed_mutex 提供。

### 3.3. Q3: std::shared_lock 能用于普通的 std::mutex 吗？

+ 答案：绝对不能。
  + 编译器会报错。因为 std::mutex 没有实现 lock_shared()、unlock_shared() 等成员函数。
  + ```std::shared_lock``` 模板要求传入的 Mutex 类型必须满足 SharedMutex 概念（即支持共享锁定语义）。

### 3.4. Q4: C++17 的 std::scoped_lock 与 C++14 的这两个类有什么关系？

+ 答案：
  + ```std::scoped_lock``` (C++17) 主要用于一次性锁定多个互斥量以避免死锁（内部使用死锁避免算法）。
  + 它通常用于独占锁定。虽然它可以锁定 ```shared_mutex```，但它默认以独占模式锁定。
  + 如果你需要对多个读写锁进行共享模式锁定，C++17 之前（包括 C++14）仍需手动管理或使用多个 shared_lock 对象。scoped_lock 主要是为了简化 unique_lock 的多锁场景

## 4. 性能与陷阱

1. 写饥饿 (Writer Starvation)：

+ 现象：如果读请求源源不断，shared_lock 持续进入，可能导致写线程 (unique_lock) 永远无法获取锁。

+ C++14 的表现：标准的 shared_timed_mutex 实现策略依赖于底层库（pthread 或 Windows SRW Lock）。有些实现倾向于读者优先，有些倾向于公平。

+ 解决：如果需要严格的公平性，可能需要应用层逻辑控制（如限制并发读数量），或者在 C++17/20 中寻找更具体的实现保证。

2. 开销问题：

+ 读写锁的内部状态维护比简单的 std::mutex 复杂（需要记录读者计数、等待写者等）。

+ 结论：只有在读操作非常频繁且耗时，而写操作很少的场景下，使用 shared_timed_mutex 才有性能收益。如果写操作也很频繁，读写锁的上下文切换开销可能反而比单纯互斥量更大。

3. 头文件依赖：

+ 必须包含 <shared_mutex>。

+ 编译时需确保开启 C++14 标准 (-std=c++14)。某些旧编译器（如 VS2015 早期版本）可能支持不完整，需注意。
