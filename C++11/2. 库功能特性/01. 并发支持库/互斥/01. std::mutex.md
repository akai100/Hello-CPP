
```std::mutex``` 是 C++ 标准库（自 C++11 起引入，位于 <mutex> 头文件中）提供的一种互斥量（Mutex）类型，用于在多线程环境中保护共享数据，防止数据竞争（Data Race）。

## 1. 核心概念

+ 互斥

  在任何给定时刻，只有一个线程可以持有（lock）该互斥量。其他试图获取该锁的线程会被阻塞，直到锁被释放。

+ 所有权

  ```std::mutex``` 具有所有权概念。哪个线程锁住了它，就必须由哪个线程来解锁。

## 2. 基本用法

### 2.1. 手动锁定与解锁


这是最基础的用法，但不推荐直接使用，因为如果临界区代码抛出异常，```unlock()``` 可能永远不会被调用，导致死锁。

```c++
std::mutex mtx;           // 全局互斥量
int shared_data = 0;

void worker() {
    mtx.lock();           // 尝试获取锁，如果已被占用则阻塞
    // --- 临界区开始 ---
    ++shared_data;        // 安全地修改共享数据
    // --- 临界区结束 ---
    mtx.unlock();         // 释放锁
}
```

### 2.2.  推荐使用：RAII 风格 (```std::lock_guard``` 或 ```std::unique_lock```)


现代 C++ 强烈建议使用 RAII（资源获取即初始化）机制来管理锁的生命周期。当守卫对象离开作用域时，析构函数会自动释放锁，即使发生异常也能保证安全。


#### 2.2.1. 使用 std::lock_guard (最简单，性能略高)

适用于只需要简单的“进入临界区 -> 离开临界区”场景

```c++
std::mutex mtx;
int data = 0;

void safe_increment() {
    // 构造时自动 lock()
    std::lock_guard<std::mutex> guard(mtx); 
    ++data; 
    // 析构时自动 unlock()
}
```

#### 2.2.2. 使用 ```std::unique_lock``` (更灵活)

适用于需要延迟锁定、手动解锁、或者配合条件变量 (```std::condition_variable```) 使用的场景.

```c++
std::mutex mtx;

void flexible_locking() {
    std::unique_lock<std::mutex> ulk(mtx); 
    // ... 做一些事 ...
    
    ulk.unlock(); // 可以手动提前解锁
    // ... 做一些不需要锁的事 ...
    
    ulk.lock();   // 也可以重新上锁
}
```

## 3. ```std::mutex``` 的特性与限制

1. 不可复制也不可移动

```std::mutex``` 对象不能被拷贝或移动。这意味着包含 ```std::mutex``` 的类默认也会禁用拷贝构造和拷贝赋值运算符.

2. 非递归（Non-recursive）

+ 如果一个线程已经持有了某个 ```std::mutex```，它再次尝试锁定同一个 mutex 会导致未定义行为（通常是死锁或程序崩溃）

+ 如果需要递归锁定（即同一个线程多次获取同一把锁），请使用 ```std::recursive_mutex```

3. 性能

```std::mutex``` 通常映射到操作系统的原生互斥量（如 Windows 的 Critical Section/Mutex 或 Linux 的 pthread_mutex）。在无竞争情况下很快，但在高竞争下涉及内核态切换，开销较大。
