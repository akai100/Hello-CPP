考虑以下代码：

```c++
// 全局变量
bool ready = false;
int data = 0;

// 线程 A
void producer() {
    data = 42;
    ready = true;
}

// 线程 B
void consumer() {
    while (!ready) {
        // 等待
    }
    std::cout << data << "\n";
}
```

❌ 可能输出 0（而不是预期的 42）！

📌 原因：缺乏同步导致内存可见性问题 + 指令重排序

在多核系统中，有两个层面可能导致问题：

| 层面 | 问题 |
|------|------|
| 编译器优化 | 编译器可能将 `data = 42; ready = true;` 重排序为 `ready = true; data = 42;`（因为单线程下行为不变） |
| CPU 内存模型 | 即使编译器不重排，CPU 可能乱序执行写操作，且其他核心看不到最新值（缓存未同步） |

👉 结果：

线程 B 看到 ```ready == true；```

但 data 的写入尚未对线程 B 可见（仍在 CPU 缓存或写缓冲区中）；

所以读到 data == 0。

💡 这不是“概率低的小问题”——在 ARM/PowerPC 等弱内存模型架构上极易发生，x86 虽较强但仍可能因编译器重排出错。

## 2. 仅将 ready 改为 std::atomic<bool> 是否足够？

❌ 不够！

📌 为什么？

+ ```std::atomic``` 默认内存序是 ```memory_order_seq_cst```（顺序一致性），能防止重排序并保证可见性；

+ 但是：如果只改 ready 为 atomic，而 data 仍是普通变量，仍可能因重排序导致问题！
  
具体来说：

+ 编译器/CPU 仍可能将 data = 42 和 ready.store(true) 重排；

+ 或者 data = 42 的写入在 ready = true 之后才对其他线程可见。

✅ 正确做法：必须建立“happens-before”关系，确保 data = 42 先于 ready = true 对消费者可见。

## 3. 正确且高效的修复版本
   
✅ 使用 Release-Acquire 语义（比默认的 seq_cst 更高效）：

```c++
#include <atomic>
#include <iostream>

std::atomic<bool> ready{false};
int data = 0; // 普通变量即可，只要同步正确

void producer() {
    data = 42;                              // (1) 写数据
    ready.store(true, std::memory_order_release); // (2) 发布：禁止 (1) 被重排到 (2) 之后
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) { // (3) 获取：禁止后续读取重排到 (3) 之前
        // 等待
    }
    std::cout << data << "\n";              // (4) 读数据：保证看到 (1) 的结果
}
```
